# Fastify-specific contract

Rules and patterns that apply when building with Fastify 5 + `fastify-type-provider-zod`.

## ZodTypeProvider setup

Instantiate the server with `withTypeProvider<ZodTypeProvider>()` and register the compiler pair **before** any routes:

```ts
const server = Fastify(serverConfig).withTypeProvider<ZodTypeProvider>()
server.setValidatorCompiler(validatorCompiler)
server.setSerializerCompiler(serializerCompiler)
```

With this setup every `schema.body`, `schema.params`, and `schema.querystring` declared on a route is automatically validated by Zod. No manual `safeParse` or `zValidator` middleware is needed in module files.

## Route schema convention

Every route must declare its schema. Body/params/querystring are Zod schemas; response is a plain object mapping HTTP status codes to Zod schemas:

```ts
server.route<{ Body: ICreate<X>Body; Params: IGet<X>Params; Reply: ICreate<X>Res }>({
  method: 'POST',
  url: '/<x>',
  schema: {
    tags: ['<X>'],
    description: 'Create a <x>',
    params: SGet<X>Params,           // z.object(...)  — validated automatically
    body: SCreate<X>Body,            // z.object(...)  — validated automatically
    response: SCreate<X>Res,         // { 200: SFooData, 400: SBadRequestRes, 500: SInternalErrorRes }
  },
  preHandler: [server.authenticate],
  handler: (req, reply) => <x>Service.create(server, req, reply),
})
```

The `response` map is NOT a Zod schema — it is a plain object. The type generic `Reply` should be the inferred type of the **200 data schema** (e.g. `z.infer<typeof S<X>Data>`), not of the whole response map.

## Auth plugin decoration

Auth guards are Fastify decorators, not middleware. The auth plugin must be wrapped with `fp` from `fastify-plugin` so decorators are visible across all registered plugins:

```ts
// auth.plugin.ts
const authPlugin: FastifyPluginAsync = async (fastify) => {
  fastify.decorate('authenticate', async (request, reply) => {
    // validate session; set request.user; return 401 on failure
  })
}
export default fp(authPlugin)
```

Without `fp()`, decorators are scoped to the child context where the plugin is registered and are invisible elsewhere — routes calling `server.authenticate` will fail at runtime with "not a function".

TypeScript must know about the decorator. Declare it via module augmentation in `auth.interface.ts`:

```ts
declare module 'fastify' {
  interface FastifyInstance {
    authenticate: (request: FastifyRequest, reply: FastifyReply) => Promise<void>
    authenticateAdmin: (request: FastifyRequest, reply: FastifyReply) => Promise<void>
  }
  interface FastifyRequest {
    user?: I<User>
    session?: I<Session>
    adminUser?: I<AdminUser>
  }
}
```

Import `auth.interface.ts` once (in `server.ts` or a global `types.d.ts`) to activate the augmentation.

## Fastify plugin pattern (fp)

Any `pkg/<name>/` slot that decorates the server or modifies the global request/reply context **must** be wrapped with `fp`. Plugins that only register hooks or routes scoped to a child context don't need `fp`. Rule of thumb: if a module or another plugin needs to call `server.<decorator>`, the defining plugin needs `fp`.

## Error response shapes

Define common error schemas once in `entities/dto/common.dto.ts` and reuse in every response map:

```ts
export const SBadRequestRes = z.object({ error: z.string(), message: z.string().optional() })
export const SUnauthorizedRes = z.object({ error: z.string(), message: z.string().optional() })
export const SNotFoundRes = z.object({ error: z.string(), message: z.string().optional() })
export const SInternalErrorRes = z.object({ error: z.string() })
```

Services always return:
- `reply.code(400).send({ error: 'Bad Request', message: '...' })` for validation failures caught manually
- `reply.code(401).send({ error: 'Unauthorized', message: '...' })` for auth failures
- `reply.code(404).send({ error: 'Not Found', message: '...' })` for missing resources
- `reply.code(500).send({ error: 'Internal Server Error' })` inside every `catch` block

## Caching

HTTP cache control headers are set in the service or a shared helper:

```ts
// set CDN/browser cache headers — call inside a service method before send
reply.header('Cache-Control', `public, max-age=0, s-maxage=${ttl}`)
reply.header('Cache-Tag', tag)  // for CDN tag-based purge
```

For server-side caching, `fastify-cacheman` provides Redis (when `REDIS_URL` is set) or in-process memory fallback. Configure it once at server startup and access via the decorated `server.cache` if needed.

There is no `waitUntil` mechanism in Fastify — cache invalidation happens synchronously in the request handler, or is triggered via a background job outside the request lifecycle.

## Payload CMS types

When using Payload CMS, `payload-types.ts` is generated by `yarn generate:types`. Run this command after every change to a collection definition. Never hand-edit the file — the next generation overwrites all manual changes.

## Env access

Read environment variables only through `envConfig` from `src/config/env.config.ts`. Add new vars to the Zod schema **and** document them in `.env.example`. Never read `process.env.<X>` directly outside `env.config.ts`.

## Secrets

Use your secret manager (Infisical, Vault, AWS Secrets Manager, etc.) to inject secrets at runtime. Never commit secret values to config files. Local development reads from `.env` (git-ignored); keep `.env.example` updated.
