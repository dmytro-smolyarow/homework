# Pitfalls & verification

## Common mistakes

**Layer dependency direction.** A feature importing a widget, an entity importing a feature, or a module importing another module are all upward imports. Lift the shared symbol down: shared types into `shared/interface/`, shared logic into a feature/widget, shared persistence into `entities/`.

**Missing `fp()` on auth plugin.** Defining decorators in a Fastify plugin without wrapping with `fp` from `fastify-plugin`. Without `fp`, decorators are scoped to the child context and invisible to the rest of the server. Routes calling `server.authenticate` will fail at runtime with "not a function".

**Business logic in `*.module.ts`.** Module files register routes and delegate — nothing else. Move all domain work to `*.service.ts`. A module method body should be: declare schema, set preHandlers, call `service.method(server, req, reply)`.

**Skipping `schema.response` on a route.** The ZodTypeProvider uses the response map to serialize the reply. Omitting it means responses are not validated or serialized. TypeScript also loses the `Reply` generic. Always declare the full response map.

**Response schema as a Zod object.** Response schemas are plain objects `{ 200: SFooData, 400: SBadRequestRes, ... }`, not `z.object({ ... })`. Using `z.object()` as the response breaks the status-code mapping and produces incorrect serialization.

**Hand-editing `payload-types.ts`.** When using Payload CMS, this file is generated by `yarn generate:types`. Edits are overwritten on the next regeneration.

**Direct `process.env` access.** Read env variables only through `envConfig`. Adding a var means updating the Zod schema in `src/config/env.config.ts` *and* `.env.example`.

**`pkg/*` reaches into `app/*` (or another `pkg/*`).** A `pkg/*` slot must be liftable as one folder into another project; once it imports from `app/*` or a sibling `pkg/*`, that property is gone. If a pkg needs a project-specific shape, accept it as a parameter at call time. If two pkg slots need the same helper, duplicate it as a private file inside each.

**Module imports another module.** Modules don't import each other. If two modules need the same logic, lift it down into a feature; if they need the same shape, lift it down into `shared/interface/`.

**Missing barrels (or stray layer-level barrels).** Slice and segment folders ship an `index.ts`; consumers import from the slice/segment folder. Layer-level folders (`modules/`, `widget/`, `features/`, `entities/`, `shared/`, `pkg/`) do **not** carry a barrel.

**Plural `widgets/` folder.** The widget layer folder is singular: `app/widget/`. Using `app/widgets/` diverges from the project convention and breaks layer-level grep checks.

**Mixed concerns in `shared/`.** A util that calls a service is no longer a util — move it to `shared/service/`. A constant file that imports runtime code is not a constant — split it.

**Folder naming with camelCase or underscores.** All folders are `kebab-case`. The slice folder name matches the file prefix: `<module>/<module>.module.ts`.
